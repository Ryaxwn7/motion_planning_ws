Topic: Hybrid architecture plan - host chooses gather center/shape, robots execute local navigation and shape control

Decisions
- Host remains responsible for strategic planning only: choose gather center, choose shape type, choose shape heading, monitor progress, and issue replan/task updates.
- Each robot must run its own `move_base` and its own `shape_assembly` loop locally to preserve control frequency and scale to larger formations.
- Host should no longer directly send per-robot `move_base` goals as the main online path.
- Each robot should instead receive a shared formation task and compute its own local navigation target near the shape.

Recommended Runtime Split
- Host: ROS master, map server, fm2_gather-based supervisor, optional map combine from robot costmaps, task publisher, monitoring/replan decision.
- Robot: base driver, lidar driver, robot_pose_ekf, AMCL, move_base, local/global costmaps, my_planner, local shape_assembly agent, local task-to-goal bridge.

Protocol Direction
- Host publishes a shared formation task containing center, heading, shape type, and task id/version.
- Robots use the shared task to generate local staging/navigation goals and later switch into distributed shape control.
- Robots report navigation/formation state back to host through topics, not ROS params, for scalable monitoring.
- Robots continue using local sensors and local costmaps; host may consume low-rate costmap/costmap_update data for strategic replanning.

Open Follow-up
- Add a dedicated `ShapeTask` message and a local per-robot task bridge.
- Convert host monitor path from direct goal dispatch to task-level replan only.
- Rework current `/shape_assembly/active_robot_ids` parameter path into topic-based status reporting for large-scale operation.
