Topic: Replanned deployment for decentralized per-robot move_base and shape_assembly

Decisions
- Updated the deployment recommendation from "host-heavy" to "robot-local control loops".
- New priority is: each robot runs its own `move_base` stack and its own `shape_assembly` controller to avoid host-side control-frequency bottlenecks.
- Host should be removed from the runtime control loop as much as possible and only keep low-frequency orchestration.
- `fm2_gather` and `c_plan_monitor` are no longer recommended as the main online control path for large-scale deployment.

Recommended Runtime Split
- Robot side: base driver, lidar driver, TF/ekf, AMCL, move_base, local planners, costmap cleaner, local shape_assembly agent, and a local bridge from shared center goal to local move_base goal.
- Host side: ROS master, map server (optional central), operator UI / task publisher, RViz / logging, and optional monitoring only.

Protocol Direction
- Host broadcasts a low-frequency shared formation center / task.
- Each robot computes its own local pre-shape navigation goal and local takeover condition.
- Robot-to-robot coordination remains distributed through shared odom / local neighbor filtering.
- Global ROS params like `/shape_assembly/active_robot_ids` should not remain in the control-critical path for the decentralized design.

Open Follow-up
- Split current launch chain into decentralized robot-side launch plus lightweight host-side task launch.
- Replace centralized gather logic with a local per-robot center-to-goal bridge.
- If scaling further, reduce all-to-all odom fanout or move to neighbor-only communication.
